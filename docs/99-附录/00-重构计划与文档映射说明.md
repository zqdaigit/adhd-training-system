## 2. 本轮重构目标

本轮重构的目标不是简单整理旧文件，而是基于现有资料，在**保持原有业务意图**的前提下，重新建立一套适用于综合系统、系统域、多软件协同场景的产品需求文档体系。

重构重点包括：

- 从“零散旧文档”升级为“分层的新文档体系”；
- 从“按历史目录堆叠”升级为“按产品需求层级组织”；
- 从“单篇说明混合多种内容”升级为“总纲、系统域、软件、专题分层表达”；
- 从“难以定位旧内容归属”升级为“新旧文档具备清晰映射关系”。

## 6. 新旧文档之间的参考关系

本轮重构不是对旧文档进行“原样搬运”，而是对旧文档进行“提炼—拆分—归位—重写”。

### 6.1 参考原则
后续所有新文档编写，均应遵循以下参考原则：

1. **旧文档提供业务原意和历史线索**  
   旧文档是业务背景、历史规划、原有功能分布和既有表达方式的重要来源。

2. **新文档负责重构表达与重新归位**  
   新文档不按旧目录生搬硬套，而是按照新的分层结构重新组织内容。

3. **保持原意，不等于保留原结构**  
   对旧文档中反映出的业务目标、系统职责、功能归属、流程逻辑应尽量保持原意；但对于旧目录结构、旧命名方式、混杂表述，可进行重构和标准化。

4. **多份旧文档可共同支撑一份新文档**  
   一份新文档可能同时参考多份旧文档，不要求一一对应。

5. **一份旧文档也可能拆入多份新文档**  
   特别是总体类、流程类、部署类旧文档，常常同时包含背景、结构、流程、职责等多种信息，后续会拆散归入不同新文档。

## 9. 编写约束说明

后续编写所有新文档时，统一遵循以下约束：

1. 仅撰写**产品需求**内容；
2. 不撰写界面设计、技术架构、代码、算法、测试、部署、运维相关内容；
3. 优先保留旧文档中的业务原意、系统关系和功能归属；
4. 可以重构旧目录、旧命名、旧表达方式，以提升整体清晰度；
5. 每完成一份文档，应尽量单独提交 GitHub，并向用户确认提交成功；
6. GitHub 相关操作默认按 `github-ops` 方式处理。
